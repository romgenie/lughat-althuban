# ar-v1 dictionary review — 2026-04-19

**Reviewer**: Claude (planner)
**Status**: planner-only findings — no changes applied; user decides what acts on what

This is a sanity-pass review of `dictionaries/ar-v1.md` against the live
loaded dictionary, ADR 0003 (governance), ADR 0004 (normalization), and ADR
0008 § B.0 (dictionary frozen at Phase A ship). Findings are split by what
action category they fall into, since the dictionary is now frozen for any
change that would alter an existing entry's semantics.

**Reviewer caveat:** I am not a native Arabic speaker. Findings about
*structure*, *consistency*, *normalizer interaction*, *Python coverage*,
and *collision/count math* are confident. Findings about whether a
specific Arabic word is *idiomatic* or *natural-sounding to a learner* are
flagged as "needs native-speaker eye" and left for you.

## Category B — visible-vs-stored form rendering (cosmetic, fix-in-place)

The dictionary mixes two rendering conventions for entries containing
characters that the ADR 0004 normalizer folds:

- **Pre-normalization (visible) form**: `إذا`, `أكد`, `وإلا`, `دالة`
- **Post-normalization (stored) form**: `اي` for `any`, `خطا` for `False`,
  `اكد` for `assert`'s rationale section

This is internally inconsistent and confusing. The reader can't tell
without checking the normalizer whether `أكد` and `اكد` are the same
entry. Recommend: pick one convention and stick to it across the file.
Two defensible options:

- **Show visible form** (`أكد`, `إذا`, `دالة`) — what the learner will type,
  more pedagogically friendly. Add a one-line note: "stored form folds
  to X per ADR 0004."
- **Show stored form** (`اكد`, `اذا`, `داله`) — what error messages will
  print, what `dialect.names` actually contains. More technically honest
  but visually jarring for an Arabic reader.

Strong recommendation: **visible form**, since this is the canonical
human-readable doc and learners will copy from it. Today's mix-and-match
is the worst of both — one canonical column with two conventions.

This is cosmetic (no semantic change), so it's allowed under "frozen
dictionary" rules.

## Category C — coverage gaps (additions, allowed under ADR 0003)

ADR 0003 says "additions for Python features not previously covered do
not require an ADR." These are missed-coverage items, not redefinitions.

### C1. `breakpoint()` missing

Standard 3.7+ debugger entry. Beginners encounter this in tutorials.
Suggest `نقطة_توقف` ("stopping point") — but defer to your judgment.

### C2. Set methods entirely missing

Section 6 has String, List, Dict, and Generic methods. Set methods
(`.add`, `.union`, `.intersection`, `.difference`, `.discard`, etc.) are
absent. If sets are part of Phase A pedagogy (and Packet 0010's
`05_data_structures.apy` does use sets indirectly via `dict`), this is a
real gap. If sets are deferred to Phase B, the dictionary should say so
in "Known omissions."

### C3. Common string methods missing

`.title`, `.capitalize`, `.swapcase`, `.zfill`, `.center`, `.ljust`,
`.rjust` are not covered. `.title()` and `.capitalize()` show up in many
beginner tutorials. Worth a v1.1 batch.

### C4. Dict `.pop` and `.popitem` missing

Both are common in beginner code. List has `.pop`; dict doesn't (in this
dictionary). Either add or note "dict.pop is method-position; resolves
via dialect.attributes lookup of `انتزع` if the user spells it the same
as list.pop." (Actually that's how it would already work — both lists
and dicts have `.pop` in real Python and the dialect maps `.pop → انتزع`
once. So this might be a doc gap, not a code gap. Worth verifying.)

### C5. `aiter`, `anext`

Async iterator builtins (3.10+). If async is targeted at Phase B per
"Known omissions," fine — leave as is.

## Category D — entries worth a native-speaker re-look (would require ar-v2)

These are choices that a native-speaker learner-tester might push back
on. Each requires `ar-v2` + a superseding ADR to change. Listed here for
your judgment, not as a recommendation to change.

### D1. `is` → `هو`

`هو` is the third-person singular masculine pronoun ("he/it"). It's one
of the most common words in Arabic. A learner trying to write a variable
named `هو` (e.g., translating `he = Person(...)`) gets a `SyntaxError`
because it's a reserved word. Hedy's choice for `is` would be worth
checking; alternative `يكون` ("to be") is listed but rejected with no
rationale.

### D2. `as` → `كـ`

The kaf-with-tatweel form `كـ` normalizes to single character `ك` after
ADR 0004 strips the tatweel. A single-character keyword is unusual and
collides easily — e.g., a variable `ك` (uncommon but legal). And
visually, `استورد طلبات كـ ط` shows learners a one-character keyword
while every other keyword is multi-character. Defensible but worth
re-examining; alternative `باسم` ("by-the-name-of") is more verbose but
unambiguous.

### D3. `enumerate` → `رقم`

`رقم` is "number" (noun). Python's `enumerate` is closer to "with-index"
or "numbered". A learner reading `لكل (i, x) في رقم(items)` might parse
it as "for each in NUMBER(items)" rather than "for each in
NUMBERED(items)". The alternate `عدد_متسلسل` ("sequential number") is
more semantic but verbose. Worth re-checking against Hedy.

### D4. `iter` → `كرر`

`كرر` is the imperative "repeat (it)". Python's `iter()` makes an
iterator, doesn't repeat. Closer translations: `كرّر_عبر` ("iterate
through"), `مكرر` ("iterator/repeater"). Minor semantic gap.

### D5. Imperative-vs-noun method-name inconsistency

Methods are split across two grammatical forms with no documented rule:

- **Imperative** (most): `.append → اضف`, `.find → ابحث`, `.replace → استبدل`,
  `.copy → انسخ`, `.clear → امسح`, `.reverse → اعكس`, `.insert → ادرج`,
  `.remove → ازل`, `.pop → انتزع`, `.update → حدث`
- **Verbal noun / non-imperative**: `.split → قسم`, `.lower → صغير`,
  `.upper → كبير`, `.strip → جرد`, `.count → عد`, `.join → اجمع`

This is typical of bilingual dictionaries (translators code-switch
unconsciously) but inconsistent. Rule of thumb in CS-Arabic translation
is to use either consistent imperatives ("do X to the receiver") or
consistent verbal nouns ("X-ing"). Worth deciding for ar-v2.

### D6. Definite-article inconsistency

Some entries carry the definite article `ال`, others don't, with no
documented rule:

- With `ال`: `.next → التالي`, `super → الاصل`, `max → الاكبر`,
  `min → الاصغر`
- Without `ال`: `len → طول`, `abs → مطلق`, `len → طول`

Pattern *might* be: superlatives (`max`, `min`) and ordinals (`next`,
`super`) take `ال` because Arabic grammar prefers them definite; plain
nouns don't. If so, document the rule. If not, standardize.

### D7. `lambda` → `لامدا`

Pure transliteration. Defensible (mathematics uses Greek λ for lambda
worldwide), but feels lazy in a dictionary that otherwise translates
semantically. Alternate `دالة_مجهولة` ("anonymous function") is more
pedagogical for a learner who hasn't met λ in math.

## Summary recommendation

**Phase A wrap-acceptable as-is.** None of the findings above are bugs in
the running code — the dictionary works, all 173 entries load, the test
suite is green. Treat this review as a punch-list of doc-hygiene items
(Category A, B) you can fix in-place under the "frozen" rule, plus a set
of items to remember when you eventually charter `ar-v2` (Category D).

**Suggested immediate fixes (Category A + B, ~30 min of edits):**
1. Update the stale header (A1).
2. Reconcile the count (A2) — likely just an arithmetic correction.
3. Decide and apply one rendering convention (B) across all entries.
4. Either delete or stub `ar-v1-alternates.md` (A3).
5. Update `dictionaries/README.md`'s phase-ownership line (A4).

**Suggested v1.1 additions (Category C, requires no ADR per 0003):**
6. `breakpoint()`.
7. Set methods (or document the deferral).
8. Common string methods (`title`, `capitalize`, etc.).

**Suggested ar-v2 charter inputs (Category D, deferred until next dictionary version):**
- Re-evaluate `is → هو` collision risk.
- Re-evaluate `as → كـ` single-char keyword.
- Standardize method imperative-vs-noun choice.
- Document or eliminate definite-article rule.
- Consider `lambda → دالة_مجهولة`.

None of Category D blocks Phase A or Phase B's first packet (B-001 alias
runtime per ADR 0008 § B.2).
